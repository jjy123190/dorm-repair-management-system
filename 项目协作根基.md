# 项目协作根基

## 1. 当前主线

- 项目名称：宿舍报修与工单管理系统
- 当前技术路线：`JavaFX + MyBatis + MySQL`
- 当前目标：先把桌面端主链跑稳，再继续补登录、上传、基础资料维护、统计导出
- 当前分支主线：不再回到普通 `Spring Boot Web CRUD`，桌面端就是后续协作基准

## 3. 代码分层不要打乱

- `com.scau.dormrepair.ui`
  - 只管 JavaFX 页面、交互事件、表格展示、弹窗调用
- `com.scau.dormrepair.ui.module`
  - 每个角色模块各自维护自己的页面
  - 工作台公共壳层优先复用 `AbstractWorkbenchModule` 里的共享方法，不要在学生页、管理员页、维修页各自复制一套比例布局、字段块和摘要卡
- `com.scau.dormrepair.service`
  - 定义业务能力
- `com.scau.dormrepair.service.impl`
  - 负责状态流转、事务、业务校验
- `com.scau.dormrepair.mapper`
  - 只写 SQL 和结果映射，不塞业务判断
- `com.scau.dormrepair.domain.command`
  - UI 调 service 时传入的命令对象
- `com.scau.dormrepair.domain.entity`
  - 表对应实体
- `com.scau.dormrepair.domain.view`
  - 工作台表格和统计面板展示对象
- `com.scau.dormrepair.common`
  - 应用上下文、演示账号、编号生成、通用执行器

## 4. 必须延续的界面规则

- 主窗口继续使用 `vfx` 风格窗口壳，不退回系统默认白框思路
- 登录页和工作台首页要有“桌面工作台”质感，不要做成普通表单堆砌页
- 主要页面默认一屏可看，不依赖滚轮浏览主内容
- 组件打开、聚焦、切页时不允许明显乱动
- 文本不能顶边、不能出框、不能把相邻卡片挤坏
- 错误提示统一走 `UiAlerts`，不要再用默认 `Alert`
- 壳层只讲身份、模块和动作，模块说明留在业务页里，不要头部、侧边栏、正文重复讲同一段话
- 右上 header summary 必须是固定高度的静态区域，只保留角色、身份、当前模块和退出动作，不要再塞会随模块变化的多行说明

## 5. 已经明确的协作规则

- 注释写真实逻辑，不写模板注释
- 业务代码优先保持短链路：`ui -> service -> mapper -> sql`
- 新功能先补 `service` 和 `mapper`，再补 `UI`
- 页面里需要演示角色时，优先沿用 `DemoAccountDirectory`
- 本地数据库默认约定：
  - host：`127.0.0.1`
  - port：`3306`
  - database：`dorm_repair_db`
  - username：`root`
  - password：`123456`

## 7. 当前容易踩的坑

- 旧库里 `repair_requests` 可能残留早期字段，启动时已经靠 `SchemaCompatibilitySupport` 做兼容修复
- JavaFX 默认 `ComboBox` 很容易因为弹层和阴影带来“组件乱动”和“点击卡顿”的错觉，当前修复原则是：
  - 本体固定宽高
  - 布局使用固定列比
  - 尽量减掉重阴影
- 这台机器上 Maven 命令末尾常带 `The system cannot find the path specified.`，目前属于环境尾噪，不代表编译失败

## 10. 提交前最低要求

- 先本地编译通过
- 关键页面别出框、别乱动、别靠滚轮才能看完
- 弹窗和按钮风格别回退到默认控件观感
- 改数据库相关代码时，顺手检查 SQL、实体、view、service 是否同步

## 11. 协作前提

- 每次开始干活前，先执行 `git pull origin main`，确保本地是最新代码，再开始改
- 每次完成自己这轮任务后，必须尽快提交并 `push` 到远端，不允许本地积压几天不推
- `pull` 完后先读两份文档：
  - `项目协作根基.md`
  - `ABCD协作历史记录.md`
- 读文档不是走流程，目的是先确认：
  - 哪块已经有人在做
  - 哪些共享文件不能乱碰
  - 当前有哪些坑还没收掉
- 如果本地有改动不能直接 `pull`，先提交到自己的分支，或者先 `stash`，不要带着旧文件硬改

## 12. 当前协作拆分（ABCD）

### 当前已知认领

- 你本人负责 `A`
- 也就是：桌面壳层、登录页、主题、弹窗、整体视觉统一由你主导
- 队友认领 `B / C / D` 时，默认不要反向改你的 `A` 区域，除非先沟通

### A. 桌面壳层与视觉统一
- 负责人目标：把登录页、主工作台壳层、弹窗、主题色、按钮和整体观感收稳
- 允许改动：`src/main/java/com/scau/dormrepair/ui/AppShell.java`、`src/main/java/com/scau/dormrepair/ui/LoginView.java`、`src/main/java/com/scau/dormrepair/ui/component/**`、`src/main/java/com/scau/dormrepair/ui/support/UiAlerts.java`、`src/main/java/com/scau/dormrepair/ui/support/UiMotion.java`、`src/main/java/com/scau/dormrepair/ui/theme/**`、`src/main/resources/styles/app.css`
- 不要碰：报修提交、派单、维修处理的 service 和 mapper 业务逻辑
- 交付标准：一屏展示、不乱动、不出框、弹窗和顶层窗口保持统一 vfx 风格

### B. 学生报修主链
- 负责人目标：把学生发起报修、查看历史、后续图片上传这一条链真正做完整
- 允许改动：`src/main/java/com/scau/dormrepair/ui/module/StudentRepairModule.java`、`src/main/java/com/scau/dormrepair/service/RepairRequestService.java`、`src/main/java/com/scau/dormrepair/service/impl/RepairRequestServiceImpl.java`、`src/main/java/com/scau/dormrepair/mapper/RepairRequestMapper.java`、`src/main/resources/mapper/RepairRequestMapper.xml`、相关 `domain.command / domain.entity / domain.view`
- 不要碰：管理员派单、维修员处理、全局壳层样式
- 交付标准：表单校验稳定、提交成功、历史记录只看当前学生、错误提示可读

### C. 管理员 / 维修员 / 统计主链
- 负责人目标：把派单、接单、状态更新、月度统计和首页数据看板做完整
- 允许改动：`src/main/java/com/scau/dormrepair/ui/module/AdminDispatchModule.java`、`src/main/java/com/scau/dormrepair/ui/module/WorkerProcessingModule.java`、`src/main/java/com/scau/dormrepair/ui/module/DashboardModule.java`、`src/main/java/com/scau/dormrepair/ui/module/StatisticsModule.java`、`src/main/java/com/scau/dormrepair/service/WorkOrderService.java`、`src/main/java/com/scau/dormrepair/service/impl/WorkOrderServiceImpl.java`、`src/main/java/com/scau/dormrepair/mapper/WorkOrderMapper.java`、`src/main/resources/mapper/WorkOrderMapper.xml`、相关统计 `view` 对象
- 不要碰：登录页、主题样式大改、学生报修表单结构
- 交付标准：派单成功、维修员更新状态成功、首页和统计页数据口径一致

### D. 数据库 / 文档 / 配置 / 联调保障
- 负责人目标：守住公共底座，维护文档、DDL、配置、测试和协作规则
- 允许改动：`README.md`、`AGENTS.md`、`项目协作根基.md`、`ABCD协作历史记录.md`、`docs/**`、`sql/mysql/**`、`pom.xml`、`src/main/resources/application.yml`、`src/main/java/com/scau/dormrepair/common/AppContext.java`、`src/main/java/com/scau/dormrepair/common/AppSession.java`、`src/main/java/com/scau/dormrepair/common/DemoAccountDirectory.java`、`src/main/java/com/scau/dormrepair/config/**`
- 可协调改动：如果表结构或枚举要变，D 先出结论，再通知 B / C 同步改
- 交付标准：文档能指导接手，DDL 和代码口径一致，编译链和运行链可复现

## 13. 协作提交规则

- 凡是改动，必须同步写 `ABCD协作历史记录.md`
- 每次确认“这轮任务完成”后，必须立刻 `commit + push`，不 push 不算真正收工
- 日志至少写清 6 件事：
  - 时间
  - 模块归属（A / B / C / D）
  - 改了哪些文件
  - 做成了什么
  - 还有什么没做完或需要别人同步
  - 是否已经 push，以及对应提交号
- 不要多人直接往 `main` 硬推，每个人先在自己的分支做，再合到 `main`
- 推荐分支名：`feature/A-shell`、`feature/B-student`、`feature/C-workorder`、`feature/D-docs-db`
- 提交信息前缀统一：`A:`、`B:`、`C:`、`D:`，一眼就能看出是谁在改哪块
- 改共享文件前先在群里报备。共享文件主要是：`app.css`、`application.yml`、`pom.xml`、`domain.enums`、`domain.entity`、`sql/mysql/01_init_schema_v1.sql`、`sql/mysql/02_seed_demo_data.sql`
- 合并前先执行：`git pull --rebase origin main`，解决完冲突再推自己的分支
- 一次提交只做一件事，不要把 UI、SQL、文档、配置混成一个大包
- 如果必须跨块改文件，在提交说明里写清“顺手同步了哪一层，原因是什么”
