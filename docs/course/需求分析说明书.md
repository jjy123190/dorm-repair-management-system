## 1 引言

### 1.1 编写目的

本文档用于明确宿舍报修与工单管理系统在课程设计阶段的业务范围、用户角色、功能需求、用户界面需求、数据词典和运行环境约束，为后续概要设计、数据库设计、编码实现和课程答辩提供统一依据。

本文档主要读者包括：

- 课程设计小组成员
- 指导教师
- 后续负责前端、后端、数据库设计的实现者

### 1.2 背景

宿舍报修业务是学校后勤管理中的高频场景。传统线下报修存在信息记录分散、处理状态不透明、催办困难、统计效率低的问题。本项目面向学生、宿舍管理员和维修人员三类用户，建设一个基于数据库的宿舍报修与工单管理系统，支撑报修申请、工单流转、维修记录、评价反馈和月度统计等业务。

### 1.3 用户特点

- 学生用户：操作频率高，关注提交便捷、进度透明、处理结果可追踪。
- 宿舍管理员：负责审核、分类、派单、催办和统计，重视列表效率和状态可控。
- 维修人员：关注待处理工单、状态更新和维修记录填写，使用路径应短。

### 1.4 假定与约束

- 本项目课程设计阶段以数据库建模和核心业务流程实现为重点。
- 当前实现路线采用 `JavaFX 17.0.10 + MyBatis 3.5.17`，推荐运行环境为 `JDK 17`。
- 数据库采用 `MySQL 8.4 LTS`。
- 初版以桌面端工作台为主，用单机应用方式完成学生报修、管理员派单、维修处理和统计展示。
- 权限控制先按角色划分设计，后续如时间允许可接入登录鉴权。

## 2 功能需求

### 2.1 系统范围

系统面向学校宿舍管理场景，覆盖以下业务：

- 学生在线提交报修申请
- 管理员审核并派发工单
- 维修人员接单和更新维修状态
- 学生查看进度并提交评价
- 管理员查看月度统计报表
- 管理员维护宿舍基础信息

### 2.2 系统体系结构

建议采用桌面端三层结构：

- 表现层：JavaFX 工作台页面，负责表单填写、列表查询、状态展示
- 业务层：Service 层，负责业务校验、状态流转、统计处理
- 数据层：MySQL 数据库存储用户、宿舍、报修单、工单、维修记录、评价等数据

### 2.3 系统总体流程

1. 学生填写宿舍楼栋、房间号、故障类型、故障描述、联系电话和图片后提交报修。
2. 系统生成报修单，状态为“已提交”。
3. 管理员查看待处理报修单，确认后按维修人员进行派单。
4. 派单后生成工单，工单进入“已派单”状态。
5. 维修人员接单、维修、更新状态，并填写维修说明。
6. 工单完成后，报修单状态更新为“已完成”。
7. 学生查看维修结果并提交评分与评价。
8. 管理员按月份查看报修数量、完成数量、处理中数量和平均评分。

### 2.4 需求分析

#### 2.4.1 学生提交报修

**功能描述**

- 功能编号：FR-01
- 功能需求：学生提交宿舍故障报修申请，并生成报修单。

**业务建模**

- 参与者：学生
- 目标：快速提交故障信息并等待处理

**用例描述**

| 项目 | 内容 |
| --- | --- |
| 标识符 | UC-01 |
| 用例名称 | 提交报修申请 |
| 参与者 | 学生 |
| 前置条件 | 学生已进入报修页面 |
| 后置条件 | 系统新增一条报修单记录 |
| 基本流程 | 学生填写楼栋、房间号、故障类型、故障描述、联系电话、图片并提交，系统保存数据并生成报修单号 |
| 可选流程 | 输入项缺失或格式不合法时，系统提示修改后重新提交 |

#### 2.4.2 管理员审核与派单

**功能描述**

- 功能编号：FR-02
- 功能需求：管理员查看待处理报修单并创建工单派给维修人员。

| 项目 | 内容 |
| --- | --- |
| 标识符 | UC-02 |
| 用例名称 | 审核并派单 |
| 参与者 | 宿舍管理员 |
| 前置条件 | 系统中存在未完成报修单 |
| 后置条件 | 工单创建成功，报修单进入已派单状态 |
| 基本流程 | 管理员筛选待处理报修单，选定维修人员与优先级，填写派单备注并提交 |
| 可选流程 | 已完成工单或不存在的报修单不得重复派单 |

#### 2.4.3 维修人员接单与处理

**功能描述**

- 功能编号：FR-03

| 用例名称 | 更新工单状态 |
| 参与者 | 维修人员 |

| 后置条件 | 工单状态和报修单状态同步更新，生成维修记录 |
| 基本流程 | 维修人员将工单状态更新为已接单、处理中、待配件或已完成，并填写处理说明 |

## 4 非功能需求

### 4.1 性能要求

- 常规查询请求响应时间应控制在 3 秒以内。
- 报修提交、派单和状态更新操作应具备事务一致性。
- 月度统计接口在中小数据量下应可快速返回。

### 4.2 数据管理能力要求

- 系统应管理宿舍信息、用户信息、报修单、工单、维修记录和评价数据。
- 数据应支持按月份统计、按状态筛选、按报修单号追踪。
- 需要预留字段长度，避免学号、楼栋号、房间号等未来扩展时重构数据库。

### 4.3 安全保密性要求

- 不同角色仅应看到与自身业务有关的数据。
- 管理员可查看全量工单与统计。
- 维修人员仅处理派发给自己的工单。
- 学生仅查看自己提交的报修和评价入口。

### 4.4 灵活性要求

- 支持后续接入登录鉴权与权限控制。
- 支持扩展新的故障类型和报表维度。
- 支持从单宿舍楼栋扩展到多校区、多宿舍区场景。

## 5 运行环境规定

### 5.1 设备

- 开发机：普通 PC 或笔记本
- 内存：8GB 及以上
- 磁盘：至少预留 5GB 可用空间

### 5.2 支持软件

- JDK 17
- Maven 3.6.3+
- MySQL 8.4
- Windows 10/11 或常见 Linux 开发环境

### 5.3 接口

- 当前交付形态为桌面端工作台，界面层通过 `JavaFX -> Service -> Mapper` 调用链访问业务能力
- 数据访问由 MyBatis + JDBC 驱动连接 MySQL 数据库完成，不再以独立 REST 接口作为主线实现
