## 1 选题简介

宿舍报修与工单管理系统主要服务于学校宿舍管理场景，面向学生、维修人员和宿舍管理员三类用户。学生可以在线提交宿舍楼栋、房间号、故障类型、故障描述、联系电话及图片等报修信息，并查看处理进度；管理员负责审核报修、按工种分类、派发工单、催办以及查看统计报表；维修人员负责接单、处理故障、更新工单状态并提交维修说明。系统通过数据库集中存储用户、宿舍、报修单、工单、维修记录和评价反馈数据，体现事务处理、状态流转、查询统计和多角色权限控制等数据库课程核心内容。

## 2 项目目标

- 建立一个围绕宿舍报修业务的完整数据库应用场景
- 体现典型信息系统中的数据新增、删除、查询、修改能力
- 展示报修到工单再到评价的状态流转过程
- 通过月度统计展示数据库查询与聚合分析能力
- 形成需求分析、数据库设计、编码实现和报告文档的完整课程设计材料

## 3 总体设计

### 3.1 技术方案

- 桌面框架：JavaFX 17.0.10
- 开发语言：Java 17
- 数据访问：MyBatis 3.5.17
- 数据库：MySQL 8.4 LTS
- 界面形态：桌面工作台
- 连接池：HikariCP 5.1.0

### 3.2 分层结构

- UI 层：JavaFX 工作台与模块页面，负责展示和用户交互
- Service 层：编排业务流程、处理状态流转、保证事务一致性
- Mapper 层：访问数据库并编写 SQL
- Entity 层：定义持久化对象
- Command/View 层：定义界面输入对象和工作台展示对象

### 3.3 功能模块划分

- 宿舍基础信息管理
- 学生报修申请管理
- 工单派发与处理管理
- 维修记录管理
- 评价反馈管理
- 月度统计分析

## 4 数据库设计

### 4.1 概念结构设计

系统核心实体包括：

- 用户
- 宿舍
- 报修单
- 工单
- 维修记录
- 评价反馈

主要联系如下：

- 一个学生可以提交多条报修单
- 一条报修单对应一个宿舍房间
- 一条报修单最多生成一张工单
- 一张工单可产生多条维修记录
- 一条报修单最多对应一条评价反馈

### 4.2 逻辑结构设计

当前项目已设计如下主要数据表：

| 表名 | 用途 |
| --- | --- |
| `user_accounts` | 存储学生、管理员、维修人员信息 |
| `dorm_buildings` | 存储校区和楼栋基础信息 |
| `dorm_rooms` | 存储楼栋下的宿舍房间信息 |
| `repair_requests` | 存储学生提交的报修单 |
| `repair_request_images` | 存储报修图片 |
| `work_orders` | 存储管理员派发后的工单 |
| `work_order_records` | 存储工单处理过程中的维修记录 |
| `repair_feedbacks` | 存储学生评价与反馈 |

### 4.3 关键字段设计说明

| 表名 | 关键字段 | 设计说明 |
| --- | --- | --- |
| `repair_requests` | `request_no` | 业务唯一编号，便于界面展示与追踪 |
| `repair_requests` | `status` | 用于体现报修生命周期 |
| `work_orders` | `work_order_no` | 工单业务编号 |
| `work_orders` | `repair_request_id` | 与报修单建立一对一关系 |
| `work_order_records` | `work_order_id` | 与工单建立一对多关系 |
| `repair_feedbacks` | `repair_request_id` | 一条报修单仅允许一条评价 |

### 4.4 索引设计

当前已在代码中明确的索引或唯一约束包括：

- `repair_requests.requestNo` 唯一索引
- `repair_requests.status` 普通索引
- `work_orders.workOrderNo` 唯一索引
- `work_orders.repairRequestId` 唯一索引

后续可继续补充：

- `repair_requests.submittedAt`
- `repair_feedbacks.repairRequestId`
- `user_accounts.username`

### 4.5 外模式设计

从课程设计要求出发，可以抽象出以下外模式：

- 学生外模式：仅查看自己的报修、处理进度和评价入口
- 管理员外模式：查看全部报修单、工单、统计信息和宿舍基础数据
- 维修人员外模式：查看分派给自己的工单和维修记录

### 4.6 一致性维护设计

当前骨架中主要通过业务层事务维护一致性：

- 派单时同时更新工单和报修单状态
- 完工时同时更新工单完成时间和报修单完成时间
- 评价提交时限制为已完成报修单且不可重复提交

后续可扩展的数据库能力：

- 视图：面向管理员的报修-工单联合查询视图
- 存储过程：月度统计报表聚合过程
- 触发器：维护状态更新时间或约束非法删除

## 5 功能实现情况

### 5.1 已完成接口骨架

| 模块 | 接口 | 说明 |
| --- | --- | --- |
| 宿舍管理 | `POST /api/dorm-rooms` | 新增宿舍信息 |
| 宿舍管理 | `PUT /api/dorm-rooms/{id}` | 修改宿舍信息 |
| 宿舍管理 | `GET /api/dorm-rooms/{id}` | 查询单条宿舍信息 |
| 宿舍管理 | `GET /api/dorm-rooms` | 分页查询宿舍信息 |
| 宿舍管理 | `DELETE /api/dorm-rooms/{id}` | 删除宿舍信息 |
| 报修管理 | `POST /api/repair-requests` | 学生提交报修 |
| 报修管理 | `GET /api/repair-requests/{id}` | 查询报修详情 |
| 报修管理 | `GET /api/repair-requests` | 分页查询报修单 |
| 工单管理 | `POST /api/work-orders/assign` | 管理员派单 |
| 工单管理 | `PATCH /api/work-orders/{id}/status` | 维修人员更新工单状态 |
| 评价反馈 | `POST /api/work-orders/feedback` | 学生提交评价 |
| 统计分析 | `GET /api/statistics/monthly` | 查看月度统计 |

### 5.2 业务状态流转

- 报修单：`SUBMITTED -> ASSIGNED -> IN_PROGRESS -> COMPLETED`
- 异常结束：`REJECTED`、`CANCELLED`
- 工单：`ASSIGNED -> ACCEPTED -> IN_PROGRESS -> WAITING_PARTS -> COMPLETED`
