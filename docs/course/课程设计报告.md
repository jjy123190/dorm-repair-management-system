## 1 选题简介

宿舍报修是高校后勤管理中的高频业务，涉及学生报修、管理员派单、维修人员接单处理、学生确认与评价等多个环节。传统纸质登记或分散聊天记录容易出现信息重复、状态不清、统计困难等问题，因此本课题以“宿舍报修与工单管理系统”为对象，设计并实现一套基于 `MySQL` 的完整数据库应用。

当前项目已经落地为 `JavaFX + MyBatis + MySQL` 桌面工作台，不再采用早期的普通 Web CRUD 原型。系统围绕真实业务链条实现了统一登录、学生报修、管理员派单、维修员处理、工单时间线、评价反馈、宿舍目录维护、账号维护、审计日志和月度统计等能力，能够较完整地体现数据库课程设计中“需求分析、关系建模、事务控制、状态流转、统计查询、数据一致性维护”的核心要求。

## 2 项目目标

- 建立一个贴近校园后勤场景的完整数据库应用案例
- 用关系模型承载用户、宿舍、报修单、工单、过程记录、图片、评价与审计日志
- 展示“提交报修 -> 派单 -> 接单 -> 处理 -> 学生确认/返修 -> 评价”的闭环状态流转
- 体现增删改查、条件筛选、聚合统计、事务一致性和权限隔离能力
- 形成需求分析、数据库设计、系统实现、测试验证和课程答辩材料的完整交付

## 3 总体设计

### 3.1 技术方案

- 开发语言：`Java 17`
- 桌面框架：`JavaFX 17.0.10`
- 数据访问：`MyBatis 3.5.17`
- 连接池：`HikariCP 5.1.0`
- 数据库：`MySQL 8.4`
- 构建工具：`Maven 3.9+`

### 3.2 分层结构

- UI 层：JavaFX 工作台与业务模块页面，负责界面展示、交互和输入校验
- Service 层：负责业务流程、状态流转、事务边界和权限判断
- Mapper 层：负责 SQL 编写和结果映射
- Entity 层：与数据表结构对应
- Command/View 层：承载界面输入对象和工作台展示对象

这种分层方式保证了“界面不直接写 SQL、Mapper 不堆业务判断、Service 统一收口状态流转”，符合数据库课程设计中“数据层与业务层分离”的实现要求。

### 3.3 功能模块划分

当前桌面端已落地以下模块：

- 登录与账号入口：统一登录、学生自助注册、手机号找回密码、个人中心改密
- 首页概览：关键指标卡、待处理数量、近期记录摘要
- 学生报修：按宿舍区、楼栋、房间提交报修并上传现场图片
- 学生历史：查看自己的报修记录、时间线、催办、取消、补图、确认完成、申请返修、提交评价
- 管理员派单：查看待处理报修、分配维修员、跟踪工单链路
- 维修员处理：接单、处理中、提交完工说明与完工凭证
- 宿舍目录维护：维护楼栋、房间与启停用状态
- 账号维护：维护学生/管理员/维修员内部账号
- 审计日志：查看高价值操作记录
- 月度统计：查看近六个月趋势、分类占比与明细数据

## 4 数据库设计

### 4.1 概念结构设计

系统核心实体包括：

- 用户账户
- 宿舍楼栋
- 宿舍房间
- 报修单
- 报修图片
- 工单
- 工单处理记录
- 完工凭证图片
- 评价反馈
- 审计日志

主要联系如下：

- 一个学生可以提交多条报修单
- 一条报修单可关联一个宿舍房间，并保留宿舍区、楼栋、房间快照
- 一条报修单最多生成一张工单
- 一张工单可产生多条工单记录与多张完工凭证图片
- 一条报修单最多对应一条评价反馈
- 关键操作会写入审计日志，便于管理侧追溯

### 4.2 逻辑结构设计

当前 DDL 已定义如下主要数据表：

| 表名 | 用途 |
| --- | --- |
| `user_accounts` | 存储学生、管理员、维修员账号信息 |
| `dorm_buildings` | 存储宿舍区与楼栋基础数据 |
| `dorm_rooms` | 存储楼栋下房间及启停用状态 |
| `repair_requests` | 存储学生提交的报修单及位置快照 |
| `repair_request_images` | 存储报修现场图片 |
| `work_orders` | 存储管理员派发后的工单 |
| `work_order_records` | 存储派单、接单、处理中、完工、返修等过程记录 |
| `work_order_completion_images` | 存储维修完工凭证图片 |
| `repair_feedbacks` | 存储学生评价与反馈 |
| `audit_logs` | 存储高价值操作审计记录 |

### 4.3 关键字段设计说明

| 表名 | 关键字段 | 设计说明 |
| --- | --- | --- |
| `user_accounts` | `username` | 统一登录账号，系统内唯一 |
| `user_accounts` | `role_code` | 标识学生、管理员、维修员三类角色 |
| `repair_requests` | `request_no` | 报修业务编号，便于展示和追踪 |
| `repair_requests` | `student_id` | 绑定真实学生账号，历史归属按账号而不是姓名判断 |
| `repair_requests` | `dorm_area_snapshot` / `building_no_snapshot` / `room_no_snapshot` | 保留提交当时的位置快照，避免后续基础资料变更影响历史记录 |
| `work_orders` | `work_order_no` | 工单业务编号 |
| `work_orders` | `repair_request_id` | 与报修单建立一对一关系 |
| `work_order_records` | `status` | 记录链路中的关键节点，形成时间线 |
| `repair_feedbacks` | `repair_request_id` | 一条报修单仅允许一条评价 |
| `audit_logs` | `action_type` / `target_type` / `target_id` | 支撑后台审计与答辩时的可追溯说明 |

### 4.4 索引与约束设计

当前 DDL 中已明确的核心唯一约束、普通索引与外键包括：

- 唯一约束
  - `user_accounts.username`
  - `dorm_buildings (campus_name, building_no)`
  - `dorm_rooms (building_id, room_no)`
  - `repair_requests.request_no`
  - `work_orders.work_order_no`
  - `work_orders.repair_request_id`
  - `repair_feedbacks.repair_request_id`
- 普通索引
  - `repair_requests.status`
  - `repair_requests.submitted_at`
  - `repair_request_images.repair_request_id`
  - `work_orders.worker_id`
  - `work_orders.status`
  - `work_order_records.work_order_id`
  - `work_order_completion_images.work_order_id`
  - `audit_logs.operator_id`
  - `audit_logs.created_at`
- 外键关系
  - `dorm_rooms.building_id -> dorm_buildings.id`
  - `repair_requests.dorm_room_id -> dorm_rooms.id`
  - `repair_request_images.repair_request_id -> repair_requests.id`
  - `work_orders.repair_request_id -> repair_requests.id`
  - `work_order_records.work_order_id -> work_orders.id`
  - `work_order_completion_images.work_order_id -> work_orders.id`
  - `audit_logs.operator_id -> user_accounts.id`
  - `repair_feedbacks.repair_request_id -> repair_requests.id`

### 4.5 外模式设计

从课程设计角度，可以抽象出以下三类外模式：

- 学生外模式：只查看自己的报修、处理进度、补图入口、确认/返修和评价入口
- 管理员外模式：查看全部报修单、工单、宿舍目录、账号列表、统计报表和审计日志
- 维修员外模式：查看分派给自己的工单、处理时间线与完工凭证上传入口

### 4.6 一致性维护设计

系统当前通过业务层事务与约束共同维护一致性，核心规则包括：

- 学生提交报修时必须绑定有效宿舍房间，并同步写入位置快照
- 管理员派单时同时创建工单并更新报修状态
- 维修员完工后，工单进入“待学生确认”，学生确认后同单完结
- 学生申请返修时，原工单链回到返修状态，而不是新开一条报修
- 学生取消已派单报修时，服务层会同步关闭关联工单并写入工单记录
- 学生评价只能针对已完成报修，且一条报修只能评价一次
- 图片数量、归属关系、状态边界均由 Service 层统一校验

### 5.2 业务状态流转

- 报修单主状态：
  - `SUBMITTED`：学生已提交，等待管理员处理
  - `ASSIGNED` / `IN_PROGRESS`：管理员已派单，维修员处理中
  - `WAITING_CONFIRMATION`：维修员已完工，等待学生确认
  - `COMPLETED`：学生确认完成
  - `REWORK_REQUIRED`：学生申请返修，原链重新流转
  - `CANCELLED` / `REJECTED`：异常结束
- 工单主状态：
  - `ASSIGNED -> ACCEPTED -> IN_PROGRESS -> COMPLETED`
  - 若返修，则在同一条工单链路中继续记录新的处理节点

### 5.3 演示数据与初始化

项目提供了 `sql/mysql/01_init_schema_v1.sql` 和 `sql/mysql/02_demo_seed_v1.sql` 两份脚本，用于一键初始化数据库结构与最小演示数据。当前种子数据至少保证：

- 1 个启用学生账号
- 1 个启用管理员账号
- 1 个启用维修员账号
- 多条不同状态的报修单
- 已派发工单、工单记录和评价反馈示例

因此 fresh DB 初始化后，三种角色登录都能直接看到非空业务数据，便于课堂演示和答辩截图。
