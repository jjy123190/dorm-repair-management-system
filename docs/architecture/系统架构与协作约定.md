# 系统架构与协作约定

## 1. 当前目标

当前仓库不是最终成品，而是“给前端、数据库、文档三方对齐的后端起步骨架”。

## 2. 目录结构

```text
DB/
├─ AGENTS.md
├─ README.md
├─ docs/
│  ├─ api/
│  ├─ architecture/
│  ├─ course/
│  ├─ database/
│  └─ teacher-materials/
├─ sql/
│  └─ mysql/
└─ src/
   └─ main/
      ├─ java/com/scau/dormrepair/
      └─ resources/
```

## 3. 后端代码分层

```text
com.scau.dormrepair
├─ common
├─ config
├─ domain
│  ├─ entity
│  └─ enums
├─ exception
├─ repository
├─ service
│  └─ impl
└─ web
   ├─ controller
   └─ dto
```

## 4. 业务模块边界

虽然当前代码还没有完全按业务拆包，但协作时必须按下面的业务边界理解：

- 宿舍模块：宿舍楼栋、房间、床位等基础数据
- 报修模块：学生提交报修、查看报修
- 工单模块：管理员派单、维修人员更新状态、工单查询
- 评价模块：学生提交评分与评价
- 报表模块：月度统计

## 5. 协作入口

前端同事先看：

- `docs/api/接口规范.md`
- `docs/api/接口清单.md`

数据库同事先看：

- `docs/database/MySQL最小表设计.md`
- `sql/mysql/01_init_schema_v1.sql`

文档同事先看：

- `docs/course/需求分析说明书.md`
- `docs/course/课程设计报告.md`
- `docs/teacher-materials/README.md`

## 6. 接口演进规则

- 接口路径一旦给前端，就不再随意改名
- 新增字段先加文档，再改代码
- 删除字段要同步说明影响页面和数据库
- 分页结构统一返回 `PageResponse`

## 7. 数据库演进规则

- 先改 `docs/database` 和 `sql/mysql`
- 再改 Java 实体和 Repository
- 再改接口文档

## 8. 当前建议的开发顺序

1. 文档同事继续完善需求分析和课程报告
2. 数据库同事按 `sql/mysql/01_init_schema_v1.sql` 建第一版库
3. 后端同事把实体与真实表结构对齐
4. 前端同事按 `docs/api/接口清单.md` 做页面
5. 最后补鉴权、文件上传和测试
